## What went wrong (the three things)

### Issue 1 — Hash methodology was naive

**Symptom:** `pg_dump | sort | sha256sum` of the pre-drill state
differed from the post-restore state, even though row counts matched
exactly and the data was bit-identical.

**Root cause:** `pg_dump`'s output includes session-specific
metadata that varies per invocation:
- `-- Started on 2026-05-10 14:56:20 UTC` (timestamp comments)
- `SET ...` statements with cluster-state dependent values

These lines `sort` differently between runs, so the resulting hash
differs. The data itself is identical; the dump-output format isn't.

**Fix landing in next runbook revision:** verify by row-level row
count + per-table aggregation (e.g.,
`SELECT COUNT(*), MAX("createdAt") FROM student`) rather than by
dump-text hash. For a stricter check, dump with `--data-only --inserts`
and grep out comments before hashing.

**Owner / date:** project owner / next runbook revision (Phase 1.9
completion commit).

### Issue 2 — `app_user` GRANTs were lost on restore

**Symptom:** Post-restore, `pg_restore` succeeded but the application
role `app_user` could not `SELECT` from any restored table. Error:
`permission denied for table student`.

**Root cause:** `dr-restore.sh` calls `pg_restore --no-owner
--no-privileges` to keep the restored DB owned by the role currently
present in the cluster. The flags drop the GRANT statements that were
in the original migration, leaving `app_user` un-privileged on the
new database. `pg_dump` per-DB doesn't capture cluster-wide role
state by design.

**Fix during drill:** re-ran the migration's GRANT statements
manually:

```sql
GRANT USAGE ON SCHEMA public TO app_user;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO app_user;
GRANT USAGE, SELECT, UPDATE ON ALL SEQUENCES IN SCHEMA public TO app_user;
GRANT EXECUTE ON FUNCTION app.is_guardian_of(uuid, uuid) TO app_user;
```

**Permanent fix (post-drill):** two complementary changes:

1. `dr-backup.sh` should ALSO call `pg_dumpall --globals-only` and
   include the resulting SQL alongside the per-DB dumps. Restoring
   the globals re-creates roles + cluster-level grants.
2. The DR runbook gets an explicit "Step 3.5 — Re-apply GRANTs"
   section that the operator runs even when globals are restored,
   as a defense-in-depth backstop. Some grants live in the migration
   files (per-database scope) and aren't captured by `--globals-only`
   either way.

**Owner / date:** project owner / before milestone 1.9 closes.

### Issue 3 — Service restart blocked by Nx daemon stale lock

**Symptom:** After the database restore, restarting `sis-service`
hung with `Waiting for sis-service:serve:development in another nx
process` despite no other nx serve being active.

**Root cause:** Nx daemon was holding a stale lock from a prior
process that I'd killed during the cordon step. `nx reset` clears it.

**Workaround during drill:** smoke-tested via `psql` directly
(verified RLS path with app_user + tenant context). Service-level
restart verified after the timeline ended.

**Permanent fix:** runbook Step 5 ("Resume traffic") gets a
prerequisite line: `pnpm exec nx reset` before starting services if
the cordon used `kill` rather than the deploy/scale-down mechanism.
Phase 2 (k8s) replaces this with `kubectl scale --replicas=N` which
doesn't suffer the same issue.

**Owner / date:** project owner / before milestone 1.9 closes.

## Drill #2 — what we'll test that #1 didn't

The first drill validated the procedure end-to-end on the local
environment. Drill #2 escalates:

1. **Different cluster.** Restore to a fresh Postgres container
   (`sms-postgres-dr`) on the same host. Verifies the procedure
   doesn't accidentally depend on the live cluster's state.
2. **Postgres version mismatch.** Push a backup taken from
   Postgres 16 onto a 17-server. (Forward compatibility is
   guaranteed by pg_restore; the test confirms.)
3. **Larger volume.** Seed sms_sis with 100k rows + 1M outbox
   events. Measure restore RTO under realistic data shape.
4. **Encryption key custody.** Move the KEK to a different host;
   restore relying on out-of-band key transfer. Validates the
   "envelope encryption with key custody" thesis.
5. **The dashboard story.** During the restore, confirm the
   platform-overview Grafana dashboard shows the gap clearly
   (request rate drops to 0; on resume, returns to baseline).
   The observability layer's job during DR is to make the recovery
   visible to operators.

## Lessons against the milestone-doc framing

The milestone doc anticipated "every first drill finds three things
broken." This drill found **exactly three** real issues. The framing
held; the drill did what it was supposed to.

- **The procedure was useful AS-IS** (90% of the way through). The
  10% gap is what writing this post-mortem captures.
- **The verification step caught Issues 1 + 2** — Issue 1 (hash
  mismatch) was a runbook bug; Issue 2 (grants) was a real recovery
  gap. Both surfaced because the verification was real, not theatrical.
- **The cold drill was 3 minutes of actual work + ~30 seconds of
  diagnosis.** A warm drill (rehearsed beforehand) would have
  found neither issue. The cold drill IS the value.

The post-mortem itself is the artifact for the next operator —
possibly a future me who has forgotten that grants live outside the
pg_dump output. The next drill will be cheaper because of this.

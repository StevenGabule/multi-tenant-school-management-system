# Domain glossary

A running glossary for the multi-tenant SMS. Add terms as they appear in
discussions and code. Ambiguity is the enemy — when two people use the same
word for different things, this is where you reconcile.

> Update this file when you encounter a term that wasn't obvious. The cost
> is minutes; the cost of *not* doing it compounds across services.

## Tenancy & control plane

| Term | Meaning |
|---|---|
| **Tenant** | A customer of the platform (typically a school or a school district). The unit of data isolation. Every tenant-scoped row in the database carries `tenant_id`. |
| **Pool tier** | Default tier. All standard tenants share one PostgreSQL cluster per region; isolation enforced by Row-Level Security. Lowest cost; highest density. |
| **Bridge tier** | Schema-per-tenant. Used **only as a transient stop** during a Pool → Silo promotion. Not offered as a permanent product tier. |
| **Silo tier** | Premium / regulated. Each tenant gets a dedicated PostgreSQL database (and optionally a dedicated cluster). Required for ministries, large districts, and jurisdictions with strict residency laws. |
| **Control plane** | The system of record for tenants themselves: the registry, billing plans, feature flags, region pinning, lifecycle state. Lives in its own database, separate from any tenant's data plane. |
| **Data plane** | The databases that hold actual tenant data (students, grades, attendance). Subject to RLS in Pool, isolated in Silo. |
| **Tenant registry** | The single source of truth that maps a tenant to its tier, region, and DSN. Owned by the `tenant-service` (milestone 1.2). Every other service consults it on startup. |

## Architecture

| Term | Meaning |
|---|---|
| **Walking skeleton** | Alistair Cockburn's term for the thinnest end-to-end thread through every architectural layer. We build one in milestone 1.0 before adding any business feature. |
| **BFF** (Backend For Frontend) | A persona-specific backend that aggregates from internal services. Not a gateway — gateways do cross-cutting concerns; BFFs do shaping. We'll have `bff-parent`, `bff-teacher`, `bff-admin` etc. (milestone 1.7). |
| **Gateway** | In Phase 1, our single API entry point and (temporarily) host of the placeholder routes. Will become `bff-public-api` + the API gateway proper as services multiply. |
| **Bounded context** | A consistency boundary inside the domain. Each context has its own ubiquitous language. Microservices live at context boundaries — but a bounded context is a DDD concept; not every context needs its own service. |
| **Saga** | A sequence of local transactions across services, each with a compensating action. Replaces distributed transactions (which are infeasible across microservices). Enrollment is the canonical saga (milestone 1.5). |
| **Outbox pattern** | Write a domain event to a database table in the same transaction as the state change; a relay process publishes it to the message bus later. Avoids the "DB updated, event lost" failure mode. (milestone 1.4) |

## Persistence

| Term | Meaning |
|---|---|
| **RLS** (Row-Level Security) | PostgreSQL feature that enforces row visibility based on a session-level setting. We use it to ensure a query for tenant A's data **cannot** return tenant B's rows, even if the application forgets the `WHERE` clause. The unbreakable safety net. (milestone 1.1) |
| **GUC** (Grand Unified Configuration) | PostgreSQL's session/transaction-scoped config variables. We store the active `tenant_id` in `app.current_tenant_id` via `SET LOCAL` and reference it from RLS policies. |
| **Driver adapter** (Prisma) | A pluggable backend for `PrismaClient` (e.g., `@prisma/adapter-pg`, `@prisma/adapter-neon`). Prisma 7 requires one explicitly. See [ADR-0004](docs/adr/0004-prisma-7-setup.md). |
| **Prisma migration** | A versioned SQL file generated by `prisma migrate dev` from schema changes. Lives in `apps/<service>/prisma/migrations/`. Production deploys use `prisma migrate deploy` (no codegen, just SQL apply). |

## Observability

| Term | Meaning |
|---|---|
| **OpenTelemetry / OTel** | The vendor-neutral standard for traces, metrics, and logs. We export OTLP/HTTP to Jaeger locally, will export to a Collector → Tempo/Prometheus/Loki by milestone 1.8. |
| **Auto-instrumentation** | OTel modules that patch popular libraries (http, pg, @nestjs/core) at require time so spans appear without manual code. Required: instrumentation must initialize **before** any patched module is imported. |
| **Tenant baggage** | OpenTelemetry "baggage" carrying `tenant_id` across span boundaries so every span/log/metric is filterable by tenant. Non-negotiable at scale. (milestone 1.8) |
| **Liveness probe** | Kubernetes probe that asks "is the process still alive?" Failure → pod restart. **Must not** check downstream dependencies (DB, Redis); doing so cascades a transient blip into a restart storm. |
| **Readiness probe** | Kubernetes probe that asks "can this pod serve traffic right now?" Failure → traffic stops, pod is **not** restarted. Touches the database (`SELECT 1`) so a slow Postgres causes graceful degradation, not a crash loop. |

## Operations & DevOps

| Term | Meaning |
|---|---|
| **kind** (Kubernetes IN Docker) | Local Kubernetes that runs each cluster node as a Docker container. We use it for dev cluster work (milestone 1.0 step 7); replaced by managed K8s in production. |
| **`hostAliases`** | Pod-level `/etc/hosts` entries. We use them in `infra/k8s/gateway/deployment.yaml` to bridge kind pods to compose's Postgres + Jaeger by name (e.g., `postgres` → `172.17.6.3`). Brittle (IPs are dynamic); a Phase 2 ServiceWithoutSelector + Endpoints pattern replaces this. |
| **ConfigMap / Secret** | Kubernetes resources for non-sensitive / sensitive config. The gateway Deployment loads its env via `envFrom: configMapRef + secretRef`. Secrets are NEVER committed (gitignored at `infra/k8s/**/secret.yaml`); created at apply time from `.env.local`. |
| **Conventional commits** | A commit-message convention (`feat:`, `fix:`, `chore:`, `docs:`, etc.) that lets tooling derive semver bumps and changelogs. We use it for the discipline; future automation may consume it. |

## Project conventions

| Term | Meaning |
|---|---|
| **`.env.local`** | Developer-edited environment file at the repo root. Gitignored. Contains DATABASE_URL, OTLP endpoint, etc. for local dev. The pattern signals "this is local-only" even though the file would still need to be reviewed before bake. |
| **`.env.example`** | The committed template for `.env.local`. Documents every variable the project consumes. New variables get added here at the same PR they're introduced. |
| **`prisma.config.ts`** | Workspace-root Prisma 7 config. Holds `datasource.url` (moved out of schema.prisma in v7). Our scripts wrap CLI calls with `dotenv -e .env.local --` to load env vars before Prisma reads them. See [ADR-0004](docs/adr/0004-prisma-7-setup.md). |
| **Generated Prisma client** | At `apps/<service>/src/generated/prisma/`, gitignored. Recreated on every `prisma generate`. Imported via `from '../generated/prisma/client'` (not from `@prisma/client` — that's the v6 pattern). |
